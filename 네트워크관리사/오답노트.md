
ICMP 메세지를 보내는 경우는? 

```ad-claude

ICMP(Internet Control Message Protocol) 메시지가 발생하는 케이스들을 유형별로 정리해드리겠습니다:

1. 오류 보고 메시지

| 유형 | 설명 | 발생 상황 |
|------|------|------------|
| Destination Unreachable | 목적지 도달 불가 | - 네트워크 접근 불가<br>- 호스트 접근 불가<br>- 포트 접근 불가<br>- 단편화 필요하나 DF 비트 설정 |
| Source Quench | 송신 속도 제어 요청 | - 라우터/호스트의 버퍼 오버플로우<br>- 처리 용량 초과 |
| Time Exceeded | 시간 초과 | - TTL 값이 0이 됨<br>- 단편화 재조립 시간 초과 |
| Parameter Problem | 매개변수 문제 | - IP 헤더 형식 오류<br>- 필수 옵션 누락 |

1. 질의 메시지

| 유형 | 설명 | 발생 상황 |
|------|------|------------|
| Echo Request/Reply | 호스트 도달성 확인 | - ping 명령어 사용<br>- 네트워크 연결성 테스트 |
| Timestamp Request/Reply | 시간 동기화 | - 두 호스트 간 시간차 측정<br>- 네트워크 지연 측정 |
| Information Request/Reply | 네트워크 정보 요청 | - 호스트의 네트워크 주소 요청 (사용 중단됨) |
| Address Mask Request/Reply | 서브넷 마스크 요청 | - 호스트가 자신의 서브넷 마스크 확인 |

3. 라우터 관련 메시지

| 유형 | 설명 | 발생 상황 |
|------|------|------------|
| Router Advertisement | 라우터 알림 | - 라우터가 자신의 존재를 알림<br>- 라우팅 정보 브로드캐스트 |
| Router Solicitation | 라우터 요청 | - 호스트가 라우터 정보 요청<br>- 네트워크 연결 시 라우터 탐색 |
| Redirect | 경로 재설정 | - 더 나은 경로 존재 시 알림<br>- 라우팅 최적화 |

ICMP 오류 메시지 코드:

| 코드 | 의미 |
|------|------|
| 0 | 네트워크 도달 불가 |
| 1 | 호스트 도달 불가 |
| 2 | 프로토콜 도달 불가 |
| 3 | 포트 도달 불가 |
| 4 | 단편화 필요, DF 설정 |
| 5 | 소스 라우팅 실패 |

주요 특징:
4. ICMP 메시지는 IP 패킷에 캡슐화되어 전송
5. 오류 보고용 ICMP에 대한 ICMP 오류 메시지는 생성되지 않음
6. 브로드캐스트/멀티캐스트 목적지에 대한 ICMP 오류 메시지 생성 안됨
7. 단편화된 패킷의 단편에 대한 ICMP 오류 메시지 생성 안됨

보안 고려사항:
8. ICMP flood 공격 가능성
9. ping of death 방지
10. 필요한 ICMP 메시지만 허용
11. ICMP 메시지 크기 제한
12. Rate limiting 설정

```

nbstat vs netstat

```ad-claude

netstat와 nbtstat의 주요 차이점과 특징을 설명해드리겠습니다:

**NETSTAT (Network Statistics)**

주요 기능:
- 활성화된 모든 네트워크 연결 표시
- TCP/IP 네트워크 연결 상태 확인
- 라우팅 테이블 정보 표시
- 네트워크 인터페이스 통계 제공
- 포트 리스닝 상태 확인

주요 옵션:

| 옵션 | 설명 |
|------|------|
| -a | 모든 연결과 리스닝 포트 표시 |
| -n | 주소와 포트를 숫자로 표시 |
| -p | 프로세스 ID와 프로그램 이름 표시 |
| -r | 라우팅 테이블 표시 |
| -s | 프로토콜별 통계 표시 |

**NBTSTAT (NetBIOS over TCP/IP Statistics)**

주요 기능:
- NetBIOS over TCP/IP 프로토콜 통계 표시
- NetBIOS 이름 테이블 표시
- NetBIOS 이름 캐시 정보 표시
- 원격 컴퓨터의 NetBIOS 이름 조회

주요 옵션:

| 옵션 | 설명 |
|------|------|
| -a | 특정 호스트의 NetBIOS 테이블 표시 |
| -c | NetBIOS 이름 캐시 내용 표시 |
| -n | 로컬 NetBIOS 이름 표시 |
| -r | 이름 해석 통계 표시 |
| -S | 세션 테이블 표시 |

주요 차이점:

| 항목 | NETSTAT | NBTSTAT |
|------|----------|----------|
| 범위 | 모든 네트워크 연결 | NetBIOS 관련 연결만 |
| 프로토콜 | TCP/IP, UDP 등 모든 프로토콜 | NetBIOS over TCP/IP만 |
| 용도 | 전반적인 네트워크 모니터링 | NetBIOS 특화 모니터링 |
| 정보 깊이 | 포괄적인 네트워크 정보 | NetBIOS 상세 정보 |
| 활용 | 일반적인 네트워크 문제 해결 | Windows 네트워크 문제 해결 |

사용 시나리오:
1. netstat
   - 포트 스캔
   - 네트워크 연결 모니터링
   - 시스템 성능 분석
   - 네트워크 트래픽 분석

2. nbtstat
   - Windows 네트워크 이름 해석 문제
   - NetBIOS 이름 충돌 해결
   - Windows 파일 공유 문제 해결
   - NetBIOS 캐시 관리

```


# 전진에러전송 (FEC)

```ad-claude

전진 에러 수정(Forward Error Correction, FEC)은 데이터 전송 시 발생하는 오류를 수신 측에서 검출하고 수정할 수 있도록 하는 오류 제어 방식입니다.

주요 특징:
1. 송신 측에서 에러 검출 및 수정을 위한 추가 비트(중복 비트)를 전송
2. 수신 측에서 오류 발생 시 재전송 요청 없이 자체적으로 오류 수정
3. 실시간 통신이나 방송과 같은 단방향 통신에 적합

FEC의 종류:

4. 블록 코드 (Block Codes)
| 코드 종류 | 특징 |
|-----------|------|
| 해밍 코드 | - 단일 비트 오류 수정<br>- 이중 비트 오류 검출 |
| BCH 코드 | - 다중 비트 오류 수정<br>- 높은 신뢰성 |
| Reed-Solomon | - 연속된 오류(버스트 오류) 수정<br>- 디지털 방송에 널리 사용 |

5. 컨볼루션 코드 (Convolutional Codes)
| 특징 | 설명 |
|------|------|
| 연속 처리 | 비트 스트림을 연속적으로 처리 |
| 메모리 사용 | 이전 입력 비트들을 참조 |
| 적용 분야 | 무선 통신, 위성 통신 |

장점:
6. 재전송이 필요 없어 전송 지연 감소
7. 단방향 통신에서도 오류 수정 가능
8. 실시간 통신에 적합

단점:
9. 추가 비트로 인한 대역폭 사용량 증가
10. 인코딩/디코딩 과정의 복잡성
11. 하드웨어 구현 비용 증가

주요 응용 분야:
12. 디지털 방송
13. 위성 통신
14. 모바일 통신
15. 광통신
16. 데이터 저장 시스템

성능 지표:
17. 코딩 게인 (Coding Gain)
18. 코드율 (Code Rate)
19. 오류 수정 능력
20. 구현 복잡도

```
