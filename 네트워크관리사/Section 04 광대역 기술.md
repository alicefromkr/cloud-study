- 두 개 이상의 장비를 네트워크에 동시에 연결하여 X.25의 패킷 전송 기술을 고속 데이터 통신에 적합하도록 계산한 프로토콜이다.

```ad-claude
title: 프레임 릴레의 프레임 구조 

프레임 릴레이의 프레임 구조를 각 부분별로 설명해드리겠습니다:

1. Flag (플래그)

- 프레임의 시작과 끝을 알리는 표시
- 마치 편지의 봉투처럼 "여기서 시작하고 여기서 끝난다"를 알려주는 역할
- 보통 '01111110'이라는 특별한 비트 패턴을 사용

2. Address (주소)

- 데이터가 어디로 가야 하는지를 나타내는 부분
- DLCI(Data Link Connection Identifier)라고도 불림
- 마치 우편물의 배송 주소와 같은 역할

1. Information (정보)

- 실제로 전송하려는 데이터가 들어있는 부분
- 사용자가 보내고자 하는 실제 내용이 담김
- 편지로 치면 편지지에 쓴 내용에 해당

2. FCS (Frame Check Sequence)

- 전송 중에 오류가 발생했는지 확인하는 부분
- 데이터가 제대로 전달되었는지 검증하는 용도
- 마치 등기우편의 운송장 번호처럼 데이터의 무결성을 확인

3. Flag (플래그)

- 다시 프레임의 끝을 표시
- 시작 플래그와 동일한 비트 패턴 사용
- 다음 프레임과의 경계를 구분

이렇게 구성된 구조 덕분에:

- 데이터를 안전하게 전송할 수 있고
- 어디로 보내야 할지 정확히 알 수 있으며
- 전송 중 발생한 오류도 검출할 수 있습니다
```

# High-Level Data Link Control (HDLC)

- Full Duplex, Half Duplex 모두 지원하는 비트지향 프로토콜
- Point to Point 및 Multi Point 링크를 위해 ISO 에서 개발한 프로토콜 

```ad-claude
HDLC에 대해 자세히 설명해드리겠습니다!

1. HDLC의 기본 개념
- 비트 단위로 데이터를 전송하는 프로토콜
- 양방향 통신이 가능한 데이터 링크 계층의 프로토콜
- 신뢰성 있는 데이터 전송을 보장

2. 통신 방식 지원
- Full Duplex (전이중 통신)
  - 데이터를 동시에 양방향으로 주고받을 수 있음 
  - 마치 전화 통화처럼 동시에 대화 가능
- Half Duplex (반이중 통신)
  - 한 번에 한 방향으로만 통신 가능
  - 무전기처럼 번갈아가며 통신

1. 연결 구성 방식
- Point to Point (일대일 통신)
  - 두 장치간 직접 연결
  - 가장 단순한 형태의 연결
- Multi Point (일대다 통신)
  - 하나의 주 장치가 여러 부 장치와 통신
  - 네트워크 구성이 더 복잡함

2. 주요 특징
- 비트 단위 전송으로 효율적
- 오류 검출과 복구 기능 내장
- 투명성 보장 (특수 문자도 데이터로 전송 가능)
- 프레임 동기화 제공

3. 동작 모드
- Normal Response Mode (NRM)
- Asynchronous Response Mode (ARM)
- Asynchronous Balanced Mode (ABM)

실생활 비유:
- HDLC는 마치 잘 정리된 회의 시스템과 같습니다
- 여러 사람이 참여할 수 있고 (Multi Point)
- 1:1 대화도 가능하며 (Point to Point)
- 동시에 이야기하거나 (Full Duplex)
- 번갈아가며 이야기할 수도 있습니다 (Half Duplex)
```

## HDLC 오류 제어 방식 

```ad-claude
HDLC의 두 가지 주요 재전송 방식을 설명해드리겠습니다!

1. Go-back-N ARQ

- 작동 방식:
    - 오류가 발생한 프레임부터 이후의 모든 프레임을 재전송
    - 수신 측에서 오류 발견 시 해당 프레임 번호를 송신 측에 알림
    - 송신 측은 그 프레임부터 다시 전송 시작
- 예시:
    - 프레임 1,2,3,4,5를 보냈는데 프레임 3에서 오류 발생
    - 수신자가 프레임 3의 오류를 발견하고 알림
    - 송신자는 프레임 3부터 다시 시작 (3,4,5 재전송)

2. Selective Repeat(선택적) ARQ

- 작동 방식:
    - 오류가 발생한 프레임만 선택적으로 재전송
    - 다른 정상적인 프레임은 그대로 유지
    - 수신 버퍼에 정상 프레임을 저장해두고 재조립
- 예시:
    - 프레임 1,2,3,4,5를 보냈는데 프레임 3에서 오류 발생
    - 수신자가 프레임 3의 오류를 발견하고 알림
    - 송신자는 프레임 3만 재전송 (4,5는 재전송하지 않음)

비교:

1. 효율성

- Go-back-N: 불필요한 재전송이 많아 비효율적
- Selective Repeat: 필요한 프레임만 재전송해서 효율적

2. 구현 복잡도

- Go-back-N: 구현이 단순
- Selective Repeat: 버퍼 관리 등으로 구현이 복잡

1. 메모리 사용

- Go-back-N: 수신 버퍼가 적게 필요
- Selective Repeat: 많은 버퍼 공간 필요

쉬운 비유:

- Go-back-N: 책을 읽다가 3페이지에서 잘못 읽었으면 3페이지부터 다시 읽기
- Selective Repeat: 3페이지만 다시 읽고 나머지는 이미 읽은 것으로 인정
```

## HDLC 특징

```ad-claude

HDLC의 세 가지 특징을 하나씩 쉽게 설명해드릴게요!

1. "프레임 내에 제어 정보인 명령과 응답을 이용하여 연속적인 정보를 전송하는 제어 절차"
- 데이터를 보낼 때 '명령'과 '응답' 정보를 함께 실어 보냄
- 마치 대화할 때 "이해했나요?"(명령) → "네, 이해했어요"(응답) 처럼 주고받는 형식
- 이런 방식으로 끊김 없이 계속해서 데이터를 주고받을 수 있음

2. "명령은 상대방에 대한 데이터 링크의 설정, 데이터 전송, 종료 지시며, 응답은 명령에 대한 실행 결과"
- 명령의 종류:
  - "통신 시작할게요" (링크 설정)
  - "이 데이터 받아보세요" (데이터 전송)
  - "통신 끝낼게요" (종료)
- 응답의 예:
  - "네, 시작하세요"
  - "데이터 잘 받았어요"
  - "네, 통신 종료합니다"

1. "사용하는 문자 코드와 상관이 없으며 비트 삽입에 의해 투명한 데이터의 전송을 보장"
- 어떤 종류의 데이터든 전송 가능
  - 텍스트, 이미지, 프로그램 등 모든 종류 OK
- 비트 삽입으로 특수한 데이터도 안전하게 전송
  - 마치 특별한 포장 방법으로 깨지기 쉬운 물건도 안전하게 배송하는 것과 같음

실생활 비유:
- HDLC는 마치 매우 체계적인 택배 시스템과 같습니다:
  - 택배기사가 "물건 배달왔습니다"(명령) → 수령인이 "잘 받았습니다"(응답)
  - 어떤 물건이든(문자 코드 무관) 안전하게 포장해서(비트 삽입) 배달
  - 모든 과정이 체계적으로 진행되고 확인됨
```

```ad-claude
HDLC 프레임의 각 필드를 자세히 설명해드리겠습니다!

1. 시작 플래그 (8비트)

- 프레임의 시작을 알리는 특별한 비트 패턴
- 항상 '01111110' 사용
- 프레임의 경계를 구분하는 역할

2. 주소 필드 (8비트)

- 데이터를 받을 장치의 주소
- 일대일 통신에서는 보통 1바이트
- 다중점 통신에서는 확장 가능

1. 제어 필드 (8비트)

- 프레임의 종류와 기능을 나타냄
- 명령과 응답 정보 포함
- 순서 번호, 흐름 제어 정보 등 포함

2. 정보 필드 (무제한)

- 실제 전송하려는 데이터 내용
- 길이에 제한이 없음
- 없을 수도 있음 (제어 프레임의 경우)

3. FCS (Frame Check Sequence) (16비트)

- 오류 검출을 위한 체크섬
- 프레임 전송 중 오류 발생 여부 확인
- CRC(순환 중복 검사) 사용

4. 종료 플래그 (8비트)

- 프레임의 끝을 나타내는 표시
- 시작 플래그와 동일한 '01111110' 사용
- 다음 프레임과의 경계를 구분

실제 동작 예시:

5. 송신측: "여기서 시작합니다"(시작 플래그) → "2번 컴퓨터로 보내세요"(주소) → "이건 데이터입니다"(제어) → "안녕하세요"(정보) → "오류는 없나요?"(FCS) → "여기서 끝납니다"(종료 플래그)
6. 수신측: 각 필드를 순서대로 확인하면서 데이터를 올바르게 수신

이런 구조 덕분에:

- 안정적인 데이터 전송 가능
- 오류 검출과 복구 용이
- 효율적인 통신 가능
```

```ad-claude
    - 수신 확인과 재전송 요청에 사용
- 사용 예: 프레임 수신 확인, 오류 보고

1. 무번호 프레임 (U-Frame)

- 주요 기능: 연결 관리
- 특징:
    - 순서 번호 없음
    - 링크의 설정/해제에 사용
    - 모드 설정에 사용
- 사용 예:
    - 통신 시작/종료
    - 연결 상태 확인
    - 초기화 작업

각 프레임은 목적에 따라 다른 구조와 기능을 가지고 있으며, 이들이 함께 작동하여 안정적인 데이터 통신을 가능하게 합니다.
```

# ATM (Asynchronous Transfer Mode)

- 가상회선 사용하는 비동기 기술
- 첫번째 패킷이 전송 될 때 송신자와 수신자 간에 최적의 전송 경로를 확정시킨다 
- 전송 경로가 확정되면 두번째 패킷부터는 포원딩만 수행 -> 속도 빠름 
- 음성 영상 같은거 하기 좋다 
- IP 헤더를 사용하지 않고 53바이트의 고정길이 인 Cell 을 쓴다
